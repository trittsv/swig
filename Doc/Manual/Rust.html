40.1 Introduction

The Rust module exposes a conservative FFI surface. It supports primitive scalar C types, strings and other pointer values as raw pointers, the wrapper functions produced by SWIG for variables, constants, constructors, destructors and member functions, enums, and opaque proxy structs for wrapped C++ classes. It generates the usual SWIG C or C++ wrapper source file and a Rust source file named after the SWIG module, for example example.rs. Each generated C wrapper is declared as a raw Rust FFI function with a _raw suffix and also exposed through an unsafe Rust wrapper function using the SWIG symbol name.

The module also includes C++ director support modelled on the C# module. Directors allow a C++ virtual method call to dispatch back into Rust by registering Rust callback functions with the generated director object. The Rust module follows the C# module structure for wrapper naming, proxy method emission and director setup while mapping those constructs to Rust syntax.

40.2 Additional command line options

The following option is available: 

-crate <name>  Set the Rust module/crate name used in generated comments

40.3 Simple example

Given a C source file: 

int add(int a, int b) {
  return a + b;
}

and the following SWIG interface: 

%module example

%{
extern int add(int a, int b);
%}

extern int add(int a, int b);

Run SWIG and build the shared library: 

$ swig -rust example.i
$ cc -fPIC -c example.c example_wrap.c
$ cc -shared example.o example_wrap.o -o libexample.so

The generated example.rs can then be used from Rust: 

mod example;

fn main() {
    let result = unsafe { example::add(19, 23) };
    assert_eq!(result, 42);
}

Compile and run the Rust program with the generated library on the library path: 

$ rustc runme.rs -L . -l dylib=example -o runme
$ LD_LIBRARY_PATH=. ./runme

The source tree also contains the same smoke test under Examples/rust/simple. After configuring and building SWIG with Rust enabled, run: 

$ make -C Examples/rust/simple check

The Rust library follows the C# target's layering: core ABI and proxy typemaps are defined in rust.swg, pointer/reference convenience typemaps are in typemaps.i, and standard C++ library helpers are provided by std_except.i, std_string.i, std_vector.i and stl.i. C++ standard exceptions caught by std_except.i are reported to Rust as panics from the generated unsafe wrapper functions. The generated Rust code uses Rust-specific typemap names such as rusttype, rustin, rustout, rustvarin, rustvarout, rustdirectorin and rustdirectorout, corresponding to the role of C# typemaps such as cstype, csin, csout, csvarin, csvarout, csdirectorin and csdirectorout.

The Rust test-suite can be run with make check-rust-test-suite after configuring SWIG with Rust enabled. The Linux GitHub Actions workflow also runs the Rust examples and test-suite for the default compiler mode, C++11 and C++14.

40.4 Directors

Director support is enabled in the same way as other director capable target languages. Use the directors module option and apply the director feature to the C++ class whose virtual methods should be overridden from Rust: 

%module(directors="1") messages
%feature("director") Receiver;

%inline %{
class Receiver {
public:
  Receiver() : last(0) {
  }
  virtual ~Receiver() {
  }
  virtual void message(int value) {
    last = value;
  }
  int last;
};

inline void deliver(Receiver *receiver, int value) {
  receiver->message(value);
}
%}

For a class named Receiver, the generated Rust file contains an opaque Receiver proxy struct, callback type aliases such as ReceiverCallback0, a ReceiverDirector trait describing the callbacks, a raw Receiver::connect_director method, a typed Receiver::connect_director_trait helper and a ReceiverDirectorHandle helper which keeps the Rust callback object alive and disconnects the director when the handle is dropped. 

mod messages;

use messages::ReceiverDirectorHandle;
use std::os::raw::c_int;

struct MyReceiver {
    total: c_int,
}

impl messages::ReceiverDirector for MyReceiver {
    unsafe fn message(&mut self, value: c_int) {
        self.total += value;
    }
}

fn main() {
    unsafe {
        let receiver = messages::Receiver::new();
        let handle = ReceiverDirectorHandle::connect(
            &receiver,
            MyReceiver { total: 0 },
        );

        messages::deliver(receiver.as_ptr(), 42);
        assert_eq!(handle.director().total, 42);
    }
}

The exact callback alias numbers follow the order of director methods emitted for the class. For example, the first director method in Receiver uses ReceiverCallback0. The raw connect_director method remains available for code that wants to provide callback function pointers directly. The generated C++ wrapper must be compiled as C++, and the generated Rust program must link against the shared wrapper library. Director method arguments and return values are marshalled through Rust ctype, directorin, directorout, directorargout, rustdirectorin and rustdirectorout typemaps, following the same director typemap structure used by the C# module. Constructed Rust proxy objects own their C++ object by default and release it from Drop; use disown() or into_raw() when the C++ side takes ownership.
